DESIGN HANDOVER & DIRECTION

NES GAME
CREATION PLATFORM

The unified “SpriteMaker Studio” redesign — what to keep, what to fix, and how to build it properly.
For: the development team Status: direction approved · prototype only v1 · July 2026
READ THIS FIRST

The interactive prototype (“SpriteMaker Studio.dc.html”) that accompanies this document is a vision piece, not a foundation to extend. Its look, tone and overall UX approach are the approved direction. Its data model and several core mechanics are deliberately simplified and, in places, technically wrong about how the NES actually works. This document separates the two so you know exactly what to carry forward and what to rebuild.

1 · Purpose

This is a direction document, not a spec sheet. It exists because the previous build had grown into seven loosely-connected pages that pupils found confusing, and the redesign has proven that a single, game-first workspace is the right answer. That much is settled. What remains is a substantial, careful engineering effort to put a correct NES data model underneath that UX and to keep the interface calm as features grow.

Two explicit, load-bearing statements for the team:

✓ THE STYLE & APPROACH ARE WANTED

The retro-NES aesthetic, the one-screen studio, the game always running in the centre, the quest-log-instead-of-popups, and the “never lose your work” safety model are all endorsed. Build on this vocabulary — don’t re-open these decisions.

! IT IS A STARTING POINT ONLY

The current UI/UX is an early prototype. There is a lot of work to do before it behaves as it should. Expect to rebuild the graphics data model from the ground up, add the missing primitives, and actively fight clutter as you go.

2 · What to keep (the approved direction)
  • One unified workspace. Mode rail on the left, contextual dock, the game on a “TV” in the centre, quest log on the right. This replaces the seven old pages and is the core win — protect it.
  • Game-first, never a blank canvas. Booting straight into a working, editable starter game removes the “I don’t know how to start” paralysis pupils reported. Keep starters for every game type.
  • Learn-by-doing, not by pop-up. The self-ticking Quest Log and the “Needs attention” validator (with Fix → / Show me jumps) are the replacement for the intro dialogs pupils skipped. This is the right model; extend it, don’t revert to modal tutorials.
  • Progress safety. Autosave on every change, minute snapshots + snapshot-before-Play, and the Time Machine (which snapshots before restoring) directly answer “make sure work is very unlikely to be lost”. Carry this into the accounts/server-save work — never regress it.
  • NES-authentic surface. The whole UI is drawn from the 64-colour NES system palette, the CRT framing, pixel type. This is deliberate and liked. Keep it.
  • Graduated depth. Code visible one click away, Advanced mode for the pupils who already read C. The principle is right (see §6 for how to structure it).
3 · The core misunderstanding: tiles & the graphics pipeline

This is the most important section. The prototype treats a “tile” as a self-contained 16×16 block that owns its own pixels, and it treats each character as owning its own pixels too. That is not how the NES works, and it quietly makes the tool teach the wrong mental model. On real hardware, pixels live in one shared place and everything else points at them.

THE REAL NES GRAPHICS PIPELINE
1 · Pattern tables (CHR)
The ONLY place pixels exist. Two tables of 256 8×8 tiles (512 total). Each tile = 2 bits/pixel → 4 values (0–3), stored as 16 bytes. Typically one table for background, one for sprites.
→
2 · Nametable
The screen: a 32×30 grid of tile indices. Each cell just says “draw tile #57 here”. No pixels of its own — pure references into the pattern table.
→
3 · Attribute table
64 bytes that colour the screen. Palette is chosen per 2×2-tile (16×16px) quadrant — not per tile. This is why NES art has its famous colour-blocking.
4 · Palettes
1 shared backdrop + 4 background palettes + 4 sprite palettes, 3 colours each, all chosen from the 64-colour system palette. Colour 0 = backdrop / transparent.
→
5 · Sprites (OAM)
64 hardware sprites, each 8×8 or 8×16, each pointing at a pattern-table tile + one sprite palette + flip bits. Bigger characters = several sprites grouped (a “metasprite”).
→
6 · The catch
Max 8 sprites per scanline or they flicker/drop. Tile memory is tiny (256+256). Reusing tiles is the whole craft.
WHERE THE PROTOTYPE DIVERGES — FIX THESE
  • There is no way to make a single 8×8 tile. The fundamental unit of all NES art is missing. This is the headline gap: the tool cannot do the one thing the hardware is actually built around.
  • “Blocks” own their pixels. A block should be a 2×2 arrangement of references into the shared tile sheet, not a monolithic 16×16 bitmap. As built, identical bricks can’t share tiles, tile reuse is impossible, and the CHR budget is meaningless.
  • Characters own their pixels too. A character should be a metasprite: a layout of 8×8 tile references + per-sprite palette. The prototype’s “size in tiles” and “per-cell palette” are the right instincts pointing at the wrong data — they should select and colour shared tiles, not slice a private bitmap.
  • The nametable/attribute tables aren’t modelled. The world is a 16×16 grid of block letters; the real 32×30 tile grid and the 2×2-tile attribute granularity are hidden. The binary export fakes them on the way out instead of them being the source of truth.
  • Animation is stored as whole private frames. On NES, animation swaps tile indices (or CHR banks). Frames-as-copies is an acceptable beginner abstraction, but it must compile down to shared tiles, not balloon the tile budget.

The original site already had this right — it had a “Shared tileset”, a tile-0 concept, [/] tile stepping, and drag-to-swap tiles that updated every reference. The redesign dropped that foundation for monolithic blocks. The correct move is to bring the shared-tileset model back and put the new UX on top of it.

4 · The data model to build (tile-first)

Make the shared pattern table the single source of truth. Everything references it by index. This one change fixes most of §3 at once and makes the CHR budget, tile reuse and exports fall out naturally.

PatternTable { tiles: Tile[512] } // each Tile = 8×8, values 0–3
Palettes { backdrop, bg[4][3], sp[4][3] } // indices into the 64-colour NES set
Block { tiles:[tl,tr,bl,br], palette } // 2×2 REFERENCES + which BG palette
Metasprite { cells:[{tile,palette,flipH,flipV,dx,dy}], w,h }
Animation { frames:[metaspriteId], fps, tag } // walk / jump / attack / idle
Screen { nametable:[32×30 tileIdx], attr:[16×15 palIdx] }
Entity { metaspriteId, role, x, y, behaviour } // placed in a Screen
  • One editor, three lenses. The 8×8 tile editor is the primitive. The block editor and the metasprite editor are assemblers that pick tiles and palettes. The world editor stamps blocks/entities. All read/write the same tile store.
  • Edit-a-tile-changes-everywhere is a feature, not a bug — but surface it. When a shared tile is about to be edited from two places, show the “also used by…” notice the old site had, with a one-click Duplicate first.
  • Budgets become real and teachable. “CHR 214/256” now means something, because blocks and metasprites consume shared tiles. Wire this into the quest/validator layer as a positive challenge (“reuse tiles to fit the cartridge”).
  • Exports stop being fabricated. `.chr`, `.nam`, `.pal` and the cc65 C/asm all serialise the real structures directly — no on-the-way-out reconstruction.
5 · The missing primitive: an 8×8 tile editor

Add a first-class Tiles mode whose job is one thing: draw and manage the 8×8 tiles in the two pattern tables. Everything else references what’s made here.

MUST HAVE
  • A 16×16 grid of the 256 tiles per bank (background / sprite), click to select, [/] to step, arrow keys to move.
  • A large 8×8 paint canvas with the 4-value pen (0 = transparent/backdrop), previewed under a chosen palette.
  • New / duplicate / delete / clear a tile; flip H/V, rotate, copy/paste (all were in the old site — keep the shortcuts).
  • Drag-to-swap tiles that rewrites every reference, so pupils can tidy their sheet without breaking art.
  • A live “used by” readout: which blocks / metasprites / screens reference this tile.
KEEP IT FROM OVERWHELMING BEGINNERS
  • Gate it behind the “Maker/Advanced” level (see §6). Beginners paint blocks and characters; the tile sheet is there when they’re ready to see “what it’s really made of”.
  • Offer “edit the tiles of this block/sprite” as a jump-in from the block and metasprite editors, so the primitive is discoverable in context rather than as a wall of 256 squares.
  • Teach the 2×2 attribute rule at the point of colouring with the quadrant grid overlaid, not as prose.
6 · Keeping it calm: the anti-clutter strategy

The redesign fixed “too many pages” but is now trending toward “too much on one page”. The tool must serve KS2 beginners and KS5 pupils writing C. The answer is progressive disclosure, not more tabs. Adopt these as standing rules:

① EXPERTISE LEVELS

A single Beginner → Maker → Advanced switch (per pupil, teacher-settable per class). Beginner: blocks, characters, play, quests. Maker: palettes, types, tile sheet, animations. Advanced: raw C/asm, CHR banks, attribute bytes. Higher levels reveal tools; they never rearrange the ones already learned.

② CONTEXTUAL TOOLS

The stage toolbar currently shows paint/erase/palette/type/fill/copy/stamp at once. Show the common two and tuck the rest behind a “more tools” affordance. Surface a tool when its target is selected, hide it otherwise.

③ ONE JOB PER DOCK

Each mode’s dock should teach one idea. Where a dock has grown a second job (e.g. characters + background tiles crammed together), that’s a signal to split the mode or move the second job to where it belongs.

  • Default to few, reveal on demand. Every control that only Advanced pupils need should be invisible to a Year-7 on their first lesson.
  • Rules is a growing list — give it search/grouping before it becomes a wall of cards, and only show cards relevant to the current game type.
  • Measure it. “Could a confused KS3 pupil ignore this control and still finish a game?” If yes, it belongs behind a level or a disclosure.
7 · Prioritised backlog
P0 — FOUNDATIONS (do before adding features)
  • Tile-first data model with a shared pattern table (§4).
  • The 8×8 tile editor (§5).
  • Honest nametable (32×30) + attribute table (2×2-tile palette granularity), as the world’s real storage.
  • Metasprite model (OAM entries, 8×8/8×16, per-sprite palette) replacing private character bitmaps.
  • Real, round-trippable exports serialised from the above; a matching import.
P1 — CORRECTNESS & TEACHING
  • Expertise levels + progressive disclosure (§6).
  • 8-sprites-per-scanline visualiser and a live CHR-budget meter, wired into the validator as teachable limits.
  • Sprite-reactions matrix (per sprite × per tile-type: block / land / bounce / exit / ignore / call_handler) — this existed in the old behaviour page and is real depth worth restoring.
  • Accounts + server saves (the class-cloud UI is mocked today); preserve the snapshot/Time-Machine guarantees on top of it.
  • Teacher tools for real: class progress, arcade moderation queue, showcase pinning (mocked today).
P2 — REACH
  • Real game types beyond platformer: top-down, auto-runner, racer (the engine templates already exist in the old builder).
  • Scrolling worlds (multi-screen nametables) rather than discrete rooms.
  • Spawn-effect sprites on triggers; checkpoints polish; per-door destinations.
  • CHR bank switching for larger games; music import (FamiStudio) made real.
  • In-browser cc65 → .nes compile so “Download ROM” produces a real cartridge image; wire to the bundled jsnes emulator for true play-in-browser.
8 · NES constraints the tool must always enforce

These are non-negotiable truths of the hardware. If the tool lets a pupil build something that violates one, it is teaching a lie. Bake them into the model so they can’t be broken, and explain them where they bite.

Colour. 64-colour system palette; 1 backdrop + 4+4 palettes of 3. No free RGB, anywhere — including the app’s own chrome.
Tiles. 8×8, 2 bits/pixel; 256 background + 256 sprite. The budget is real and visible.
Attributes. Background palette is chosen per 2×2-tile (16×16px) block — not per tile. Show the quadrant grid.
Sprites. 64 total, 8×8 or 8×16, one palette + flip each; 8 per scanline before flicker.
Sprite colour 0 is always transparent — you cannot draw black with it. Outlines need a real colour.
Everything references tiles. Reuse is the craft; edits propagate; swaps rewrite references.
