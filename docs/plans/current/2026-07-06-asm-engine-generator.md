# ASM engine generator — through to a complete ASM rewrite — 2026-07-06

> **Scope.** How to take the pupil-facing NES engine from "6 hand-written 6502
> functions shipped, the rest in cc65 C" (engine **v19**) all the way to a
> **complete, per-project hand-written-quality 6502 engine generated by the
> server** — with no cc65 C in the shipped ROM as the end state. This is the
> "full ASM engine" backlog item (task #30) made concrete.
>
> **Governing constraint.** The engine is *generic and generated per project*.
> A pupil's world size, player size, enemy count/positions, enabled modules and
> game type are unknown until they press ▶ Play. So we cannot hand-write a whole
> game in ASM (that is what `rewrites/` does for *fixed* games — see below). We
> must **generate** the per-project engine, and prove it behaviourally
> equivalent to the C engine without hand-checking every pupil project.

## The key insight that makes this tractable

"Generate ASM per project" sounds like *emit ASM text per pupil* — a bespoke
compiler. It is **not**. Almost all per-project variation is one of:

1. **Constants** — `WORLD_COLS`, `BG_WORLD_COLS/ROWS`, `PLAYER_W/H`,
   `PLAYER_TILES_PER_FRAME`, sprite counts, start positions, speeds, deadzones,
   tile ids, palette bytes. → emit a generated **`project.inc`** of ca65
   `.define`s and let fixed ASM *reference the symbols*. ca65 bakes them at
   assemble-time, per project. (This is exactly how the shipped ASM already
   reads generated `palette_bytes` / `bg_world_tiles`.)
2. **Data tables** — behaviour map, scene-sprite arrays, nametables, palettes,
   dialogue text. → already generated today (`behaviour.c`, `scene.inc`,
   `bg_world.c`, `palettes.inc`); re-emit as ca65 `.byte` tables (the
   `main.s.starter` / `build_scene_asminc` path already does this for the raw
   path).
3. **Structural variation** — the vertical scroll path only exists if
   `BG_WORLD_ROWS > 30`; the player physics differ by `BW_GAME_STYLE`;
   `behaviour_at`'s multiply is a shift when `WORLD_COLS` is a power of two. →
   **ca65 conditional assembly** (`.if`, `.ifdef`) driven by the generated
   `project.inc`, plus per-module `.s` files the server includes only when the
   module is enabled.
4. **Counts** — N enemies, M gems. → a **generic loop** over `N` (from
   `project.inc`) reading a data table, with per-sprite behaviour dispatched on a
   type byte. Never per-entity unrolled (that is what `rewrites/` does and why it
   cannot generalize).

So the "generator" is: **generated `project.inc` (constants) + generated data
tables + conditional assembly + a library of fixed, generic, well-tested ASM
modules the server selects and includes.** The server's new job is small (emit
`project.inc`, choose which `.s` modules to include, set `.define` flags) — the
same shape as today's `builder-modules.js` C-injection system, but the modules
are hand-written ASM and the "injection" is include + define rather than text
splicing. This is a *finite, testable library*, not an open-ended code emitter.

## End state ("complete ASM rewrite")

A `/play` build in **full-ASM mode** runs `ca65` + `ld65` only — no `cc65` C
compile. `main.c`, `behaviour.c`, `scroll.c`, `bg_world.c` are gone from the
shipped path; their behaviour lives in generated `project.inc` + data tables +
the ASM module library + a generated glue main loop. Audio is already ASM
(FamiStudio). The C engine **remains in the tree** as the reference oracle and
the `PLAYGROUND_NO_ASM=1` fallback — it is never deleted, because it is how we
verify the ASM engine (see Verification).

## Phases

Each phase is independently shippable, flag-gated, golden-safe on the C path,
and behaviourally verified on the ASM path. Engine-version bump + snapshot per
phase (per `engine-versioning.md`).

### Phase 0 — DONE (engine v19)
Six project-independent functions shipped on `/play`
(`read_controller`, `write_palettes`, `world_to_screen_x/y`, `scroll_follow`,
`scroll_apply_ppu`) behind `NES_ASM_LEAF`/`NES_ASM_SCROLL`, marker-gated
(`NES_ASM_READY_V1`), `PLAYGROUND_NO_ASM` kill switch. Pure-C golden unchanged.

### Phase 1 — DONE (engine v20)
Shipped `src/project.inc` (server `build_project_inc` → ca65 `.define`s) +
`asm_macros.inc` (`MULC` shift-add). Generalised + shipped `behaviour_at`,
`reaction_for`, `advance_animation`, `scroll_stream_prepare` (conditional-assembly
row path) — 10 functions live on `/play`. Golden byte-identical, matched-progress
A/B identical, full builder suite + E2E green. Learnings: ca65 needs `.define`
(not `=`) for constants folded inside `.proc`; branch-over-`jmp` for the
variable-size (MULC/`.if`) code paths; the `platformer.c` template gates basic
anim via `NES_ASM_ANIM` (attack/racer keep inline C).

**(original goal notes below):** ship the functions that only failed because they
baked constants.
- Server emits **`src/project.inc`** from `state`: every per-project constant
  the ASM needs (`WORLD_COLS`, `BG_WORLD_COLS/ROWS`, `PLAYER_W/H`,
  `PLAYER_TILES_PER_FRAME`, `NUM_BEHAVIOUR_SPRITES`, `NUM_STATIC_SPRITES`,
  screen/deadzone constants…). Makefile passes it to `ca65` (`.include` or `-I`).
- Rewrite the four `NES_ASM_SPECIALIZED` twins to read `project.inc` symbols
  instead of literals, with `.if` for structural variants:
  `behaviour_at` (multiply/shift on `WORLD_COLS`), `reaction_for`
  (`NUM_BEHAVIOUR_SPRITES` bound), `scroll_stream_prepare` (`.if BG_WORLD_ROWS>30`
  row path), `advance_animation` (`PLAYER_TILES_PER_FRAME`).
- Fold them into the shipped flags (marker-gated). Now **10–12/12 live**.
- **Verify:** extend `asm-lab` to assemble each twin against a *range* of
  `project.inc` values (COLS ∈ {32,64,96,128}, player sizes, sprite counts) and
  A/B vs the C reference; plus the corpus dual-build (Phase 5 harness, stood up
  here first, small).
- **Risk:** low — same pattern as shipped work; the only new machinery is
  `project.inc` + conditional assembly.

### Phase 2 — the per-frame hot loops as generic ASM modules
The dominant per-frame cost (perf-budget note: "enemy AI loops dominate").
- **Scene-sprite update** (`scene_asm.s`): one generic loop over
  `NUM_STATIC_SPRITES`, reading the generated scene arrays (de-`static`'d /
  emitted as ca65 tables), dispatching AI on a per-sprite **type byte**
  (static / patrol / flyer / chaser — mirrors v10/v11), applying gravity +
  behaviour-map collision (`behaviour_at`). Replaces the inline C loop in
  `main.c` behind a flag; C stays under `#else`.
- **Player physics** (`player_asm.s`): the 5 game-type variants selected by
  `.if BW_GAME_STYLE == …` (data from `project.inc`). Reference structure from
  `rewrites/` Step 1/3.
- **OAM build** (`oam_asm.s`): generic player + scene-sprite → `$0200` shadow
  loop with `world_to_screen` (already ASM) + flip/priority.
- **Verify:** A/B matched-progress per game type + the existing render/behaviour
  builder suites (they already run through the server → now exercise the ASM).
- **Risk:** medium — this is real gameplay logic; needs the scene arrays
  de-`static`'d in the codegen (linkage-only, byte-identical, as proven for
  `palette_bytes`/`col_buf`) and input-scripted A/B (Phase 5).

### Phase 3 — the remaining subsystems as opt-in ASM modules
Mirror the C module toggles one-for-one, each a fixed `.s` the server includes
when the module is enabled (`.define`-gated, data from generated tables):
dialogue engine, doors/warps, pickups, damage/HP + i-frames, HUD (hearts),
win conditions, and the SMB cluster (powerups/blocks/pipes/flagpole/smbhud).
Plus the **vblank/NMI handler + main-loop skeleton** (`main_loop_asm.s`) that
sequences the enabled module routines — the generated "glue" (the only piece
that is assembled from a server-chosen include list rather than fixed text).
- **Verify:** per-module A/B on a project that enables exactly that module, and
  the module's existing builder-test suite.
- **Risk:** medium; breadth, not depth. Each module is small and independently
  gated, so partial completion still ships.

### Phase 4 — full-ASM mode: drop cc65 C
- Add a `NES_ASM_FULL` build mode (Makefile): assemble the generated glue +
  module library + `project.inc` + data tables, link with an ASM `crt0`/vectors
  (own it, or reuse cc65's `crt0` as the audio path already does). No `cc65`
  invocation. `main.c`/`behaviour.c`/`scroll.c`/`bg_world.c` are not compiled.
- Server exposes full-ASM mode once Phases 1–3 cover every feature a project
  uses; until then a project that ticks an unconverted module falls back to the
  C engine (or hybrid — C for that module, ASM for the rest, if linkage allows).
- **Own the ZP map + call ABI** in full-ASM mode (no cc65 fastcall/soft-stack).
- **Verify:** full corpus dual-build behavioural-equivalence + cycle/size deltas.

### Phase 5 — verification infrastructure (built incrementally from Phase 1)
The load-bearing phase; the reason this is safe. Byte-golden is impossible
(ASM ≠ C byte-for-byte) and per-pupil hand-checking is impossible, so:
- **Representative project corpus** — a fixed set of `state` fixtures: one per
  game type, plus edges (1×1, 2×2 scroll, max enemies, every module on, odd
  `WORLD_COLS`). Checked into `tools/builder-tests/asm-corpus/`.
- **Dual-build equivalence CI** — for each fixture, build the C engine
  (`PLAYGROUND_NO_ASM=1`) and the ASM engine, boot both in jsnes, run a
  **scripted input track**, and assert **matched-progress** equivalence (cam,
  WRAM game state, palette, OAM, nametables) — the A/B method extended with
  input, replacing byte-golden for the ASM path. The C path keeps its byte-
  identity golden as today.
- **Cycle/size benchmark report** — per fixture, from `.map` sizes + emulator
  cycle counts per frame (the thing `rewrites/` never produced). Proves the win
  and guards regressions.
- **Runtime safety** — `PLAYGROUND_NO_ASM` kill switch stays; the C engine is
  never removed.

## Cross-cutting concerns

- **Engine versioning.** Each phase bumps `ENGINE_VERSION` + snapshots. The
  snapshot must start capturing `project.inc`'s *generator* and the ASM module
  library. The **pure-C byte-identity golden remains the invariant** (built with
  no ASM flags); the ASM engine gets **per-corpus behavioural goldens**. Document
  in `engine-versioning.md` that a snapshot rebuilt with `make` (no flags)
  reproduces the C ROM — an accepted behaviourally-identical fallback.
- **Educational value (dual goal).** Generated ASM must be *readable* — carry the
  `rewrites/`-style `TUNABLES` + comments so a pupil can open the `.s` and learn.
  Consider a Studio "view the assembly" panel that shows the generated `.s` for
  their project (a strong teaching feature the C engine can't offer as cleanly).
- **Hybrid transition.** Until Phase 4, builds are C + ASM mixed under the cc65
  ABI (fastcall, ZP `tmp*/ptr*`, soft stack) exactly as v12–v19 do. Keep the
  marker gate so bespoke `customMainC` stays pure C.
- **Rollback.** Kill switch, per-phase flags, C engine always present, dual-build
  gate blocks a phase from shipping unless its corpus A/B is green.

## What "done" looks like

- Every `/play` platformer-family build ships a generated, per-project,
  hand-written-quality 6502 engine; no cc65 C in the ROM.
- A corpus dual-build proves behavioural equivalence to the retained C engine;
  a benchmark report shows the cycle/size win.
- The C engine remains as oracle + `PLAYGROUND_NO_ASM` fallback.
- Pupils can view the readable ASM their game compiled to.

## Sequencing / effort (rough)

Phase 1 is small and high-value (unblocks the functions already written).
Phase 5's harness is the prerequisite for trusting Phases 2–4 and should be
stood up (minimally) alongside Phase 1. Phase 2 is the big perf win (enemy
loop). Phases 3–4 are breadth + the final cutover. Estimate: Phase 1 + minimal
Phase 5 ≈ one focused effort; Phase 2 ≈ one; Phase 3 ≈ several (per module);
Phase 4 ≈ one once 3 is covered. No phase is all-or-nothing — each shippable
increment tightens the loop and can pause cleanly.
