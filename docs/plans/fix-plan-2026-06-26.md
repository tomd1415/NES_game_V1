## Plan

### Step 1 — BR-03: guard the unprotected OAM writes (the only real bug)

**File:** `tools/tile_editor_web/builder-templates/platformer.c`

The buffer is `oam_buf[256]` with `oam_idx` advancing 4 bytes per hardware sprite.
Every other writer (P2 at 1579/1611/1654, spawn at 1710, HUD hearts at 1742/1768,
scene sprites at 1948) already guards with `if (oam_idx > 252) break;`. Two writers
do not:

1. **P1 render loop (1465-1498)** — add the same guard at the top of both the outer
   (`r`) and inner (`c`) loops:
   ```c
   for (r = 0; r < PLAYER_H; r++) {
       if (oam_idx > 252) break;
       for (c = 0; c < PLAYER_W; c++) {
           if (oam_idx > 252) break;
           ...
           oam_buf[oam_idx++] = sy;   /* etc. */
       }
   }
   ```
2. **Racer lap HUD (1788-1799)** — guard before the P1 digit block and again before
   the `#if PLAYER2_ENABLED` P2 digit block:
   ```c
   if (oam_idx <= 252) { /* P1 lap digit, 4 bytes */ }
   #if PLAYER2_ENABLED
   if (oam_idx <= 252) { /* P2 lap digit, 4 bytes */ }
   #endif
   ```

**Why a guard, not a bigger buffer:** the NES PPU only displays 64 sprites; dropping
overflow sprites is the correct, idiomatic behaviour and matches the rest of the file.
Enlarging `oam_buf` past 256 would break the page-aligned `$0200` OAM DMA contract in
`cfg/nes.cfg`.

**Verify:**
- This template is consumed by the Code-page build path (`_build_in_tempdir`), so an
  end-to-end check is: generate a ROM with a maxed-out (8×8) P1 and confirm it builds
  and the player still renders. A focused build is enough — `cd steps/Step_Playground
  && make` exercises the same cc65 toolchain the server invokes.
- The golden-hash harness `tools/tile_editor_web/run-all.mjs` (referenced by commit
  `230fdce`) guards `Step_Playground` output; run it to confirm no unintended diff.

**Commit:** `fix(platformer): guard P1 + racer-HUD OAM writes against 256-byte overflow (BR-03)`

### Step 3 (optional, quality) — collapse the 3 ROM-build functions (P3)

**File:** `tools/playground_server.py`
(`_build_asm_in_tempdir`, `_build_in_shared_dir`, `_build_in_tempdir`)

All three end identically: `make` → concat stdout/stderr into `build_log` → on non-zero
`raise BuildError(build_log)` → read `game.nes` → `return rom_bytes, build_log`. Extract a
helper `_run_make_and_read(build_dir) -> (rom_bytes, build_log)` and have all three call it.
Keep the per-function asset-writing preambles as-is (they genuinely differ: shared-dir vs
temp clone vs asm).

**Caution:** `_build_in_shared_dir` holds `BUILD_LOCK` and writes to the tracked
`Step_Playground` tree; do not move that locking into the helper. This is a pure
de-dup — behaviour must stay byte-identical. Gate on `run-all.mjs` golden hashes.

**Only do this if touching the file anyway** — it's maintainability, not a bug.
